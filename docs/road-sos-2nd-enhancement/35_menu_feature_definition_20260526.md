# ROAD-SOS 메뉴별 기능정의서

작성일: 2026-05-26  
작성 기준: 2차 고도화 기획 문서, 보고서 양식 요구, 사용자 피드백, 현재 구현 상태 재점검

## 1. 문서 목적

본 문서는 ROAD-SOS 2차 고도화 범위의 전체 메뉴를 기준으로 각 메뉴의 목적, 사용자, 주요 기능, 데이터, 화면/API, 정책, 현재 구현 상태와 보완 필요사항을 정리한다.

최근 테스트 과정에서 “메인 화면만 있고 기능이 동작하지 않는 것처럼 보이는 문제”, “보고서 양식이 공통으로 나오는 문제”, “일상점검/상황일지/알림톡 수신자 관리의 실제 업무 흐름 누락”이 확인되었으므로, 이후 개발은 본 문서를 기준으로 체크하면서 진행한다.

## 3. 관리자 > 현장관리

### 3.1 목적

포트홀/도로파손 접수건을 현장 운영 데이터로 관리하고, 접수/처리/작업 상세 데이터를 기반으로 업무 보고서를 생성한다.

### 3.2 주요 사용자

| 사용자 | 역할 |
|---|---|
| 관리자 | 접수건 조회, 상세 수정, 작업 정보 보정 |
| 보고서 담당자 | 접수건 선택, 보고서 양식/형식 선택, 문서 다운로드 |

### 3.3 주요 기능

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 접수 목록 조회 | 기간, 현장, 상태, 키워드 기반 접수 조회 | 구현 |
| 접수 상세 조회 | 접수 기본정보, 위치, 사진, 처리정보 확인 | 구현 |
| 접수 상세 수정 | 작업 내용, 처리 내용, 보고서용 필드 저장 | 부분 구현 |
| 작업정보 관리 | 투입인력, 장비, 자재, 작업범위 등 구조화 입력 | 부분 구현 |
| 보고서 양식 선택 | 보고서 템플릿 선택 | 구현 |
| 출력 형식 선택 | PDF/DOCX 선택, HWPX 제외 | 구현 |
| 단건 미리보기/다운로드 | 선택한 양식으로 단건 출력 | 보정 완료 |
| 다건 선택 출력 | 체크한 접수건 기반 복합 보고서 출력 | 구현 |

### 3.4 보고서 양식

| 코드 | 양식명 | 목적 | 현재 상태 |
|---|---|---|---|
| `POTHOLE_LEDGER` | 포트홀 관리대장 | 접수별 상세 관리대장 | PDF/DOCX |
| `POTHOLE_SUMMARY` | 포트홀 집계표 | 기간별 접수/처리 집계 | PDF/DOCX |
| `MAINTENANCE_LOG` | 유지보수 일지 | 시간순 유지보수 작업 기록 | PDF/DOCX |
| `LANDSCAPE_DAILY_WORK` | 조경 작업일보 | 조경/환경 정비 작업일보 | PDF/DOCX |
| `MAINTENANCE_RESULT` | 유지관리 결과보고서 | 작업 결과 보고 | PDF/DOCX |
| `PHOTO_BOARD` | 사진대지 | 조치 전/후 사진 보고 | PDF/DOCX |

### 3.5 주요 화면/API

| 구분 | 경로 |
|---|---|
| 화면 | `/admin/ims/dashboard` |
| 목록 API | `/admin/ims/data` |
| 상세 API | `/admin/ims/detail` |
| 저장 API | `/admin/ims/save` |
| 보고서 템플릿 목록 | `/admin/reports/templates` |
| 보고서 출력 | `POST /admin/reports/export` |

### 3.6 정책

| 정책 | 내용 |
|---|---|
| 출력 형식 | 현재 기준 PDF/DOCX만 지원, HWPX 제외 |
| 양식 선택 | 보고서 생성 시 `template`, `format`, `reportNos`를 명시한다 |
| 권한 | 조회 권한이 있는 접수건만 출력되어야 한다 |
| 빈 필드 | 보고서 필수 필드 누락 시 경고 또는 빈값 표시 정책 필요 |

### 3.7 보완 필요사항

| 우선순위 | 항목 |
|---|---|
| 높음 | 보고서 원본 양식과 실제 DOCX/PDF 레이아웃 1:1 검수 |
| 높음 | 각 보고서 양식별 필수 필드 누락 경고 |
| 중간 | 공종/발생장소/작업유형 마스터 기반 선택화 |
| 중간 | 투입자원 입력 UI를 반복 행 구조로 정리 |
| 낮음 | 보고서 템플릿 관리 메뉴 별도화 |

## 4. 관리자설정 > 알림톡 수신자 관리

### 4.1 목적

알림 유형별로 발송 대상자를 관리하고, 운영자가 사용자 목록에서 수신자를 배정하거나 외부 수신자를 수동 등록할 수 있게 한다.

### 4.2 알림 유형

| 코드 | 알림 유형 | 목적 |
|---|---|---|
| `POTHOLE_RECEIPT` | 포트홀 접수 | 신규 접수 발생 시 운영 담당자 알림 |
| `POTHOLE_COMPLETE` | 포트홀 처리완료 | 작업/처리 완료 시 담당자 알림 |
| `DAILY_CHECK` | 일상점검 | 점검 제출 또는 이상 항목 발생 시 알림 |
| `SITUATION_LOG` | 상황일지 | 상황 등록/공유 필요 시 알림 |

### 4.3 주요 기능

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 알림 유형 카드 | 유형별 수신자 관리 진입 | 구현 |
| 운영 기준 안내 | 등록 정책과 발송 기준 안내 | 구현 |
| 미등록 사용자 목록 | 관리자 사용자 목록 기반 후보 조회 | 구현 |
| 등록 수신자 목록 | 선택 유형의 활성 수신자 조회 | 구현 |
| 수신자 배정 | 사용자 후보를 알림 유형 수신자로 등록 | 구현 |
| 수신자 해제 | 수신자를 미사용 처리 | 구현 |
| 수동 등록 | 사용자 목록에 없는 외부 수신자 등록 | 구현 |
| 상세 수정 | 등록된 수신자 정보 수정 | 구현 |
| 개인정보 보호 | 목록 전화번호 마스킹 | 구현 |
| 발송 연동 | 실제 외부 알림톡 발송 | 제외/로컬 스텁 |

### 4.4 주요 화면/API

| 구분 | 경로 |
|---|---|
| 화면 | `/admin/notification/recipients` |
| 수신자 목록 | `GET /admin/notification/recipients/data` |
| 수신자 상세 | `GET /admin/notification/recipients/{recipientId}` |
| 수신자 저장 | `POST /admin/notification/recipients/save` |
| 수신자 해제 | `POST /admin/notification/recipients/delete` |
| 사용자 후보 | `GET /admin/user/data` |

### 4.5 정책

| 정책 | 내용 |
|---|---|
| 발송 기준 | 알림 유형, 관리대상, 사용 여부가 발송 대상 판단 기준 |
| 중복 방지 | 동일 알림 유형/관리대상/사용자 재등록 시 기존 미사용 row를 재활성화 |
| 삭제 방식 | 물리 삭제 대신 `use_yn = 'N'` 처리 |
| 로컬 환경 | 실제 발송 없이 저장/조회/로그 흐름만 검증 |

### 4.6 보완 필요사항

| 우선순위 | 항목 |
|---|---|
| 중간 | 알림톡 발송 이력 화면 |
| 중간 | 발송 실패/재시도 정책 |
| 중간 | 알림 유형별 메시지 템플릿 관리 |
| 낮음 | 부서/역할별 자동 추천 수신자 |

## 5. 관리자설정 > 일상점검 체크리스트

### 5.1 목적

관리자가 점검 유형과 체크사항, 답변 옵션을 등록하여 현장 사용자가 동일 기준으로 일상점검을 수행할 수 있게 한다.

### 5.2 주요 기능

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 체크리스트 목록 조회 | 현장/사용 여부/키워드 기반 조회 | 구현 |
| 체크리스트 등록 | 체크리스트명, 현장, 설명, 사용 여부 저장 | 구현 |
| 체크리스트 수정 | 기존 체크리스트 정보 변경 | 구현 |
| 체크리스트 삭제 | 미사용 또는 삭제 처리 | 구현 |
| 항목 등록 | 유형, 체크사항, 입력형식, 필수 여부, 정렬 저장 | 구현 |
| 답변 옵션 등록 | 양호/주의/불량 등 항목별 답변 옵션 저장 | 보정 완료 |
| 사용자 화면 반영 | 활성 체크리스트를 현장 사용자에게 노출 | 구현 |

### 5.3 주요 화면/API

| 구분 | 경로 |
|---|---|
| 화면 | `/admin/daily-checklists/setting` |
| 목록 | `GET /admin/daily-checklists/data` |
| 상세 | `GET /admin/daily-checklists/{checklistId}` |
| 저장 | `POST /admin/daily-checklists/save` |
| 삭제 | `POST /admin/daily-checklists/delete` |

### 5.4 정책

| 정책 | 내용 |
|---|---|
| 사용 여부 | 사용 중인 체크리스트만 사용자 작성 화면에 노출 |
| 필수 항목 | 필수 항목 누락 시 점검 완료 불가 |
| 답변 옵션 | 항목별 옵션을 우선 사용하고 없으면 기본 옵션 사용 |
| 현장 범위 | 현장별 또는 공통 체크리스트를 사용할 수 있어야 함 |

### 5.5 보완 필요사항

| 우선순위 | 항목 |
|---|---|
| 높음 | “유형”을 자유입력으로 둘지 코드 마스터로 둘지 확정 |
| 중간 | 체크리스트 복사/버전 관리 |
| 중간 | 점검 항목 수정 시 기존 점검 이력 보존 정책 |
| 낮음 | 엑셀 업로드 기반 항목 일괄 등록 |

## 6. 현장 사용자 > 일상점검 작성

### 6.1 목적

현장 사용자가 관리자가 등록한 체크리스트를 선택하여 매일 점검을 수행하고, 항목별 답변/메모/사진을 저장한다.

### 6.2 주요 기능

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 체크리스트 선택 | 사용 가능한 체크리스트 선택 | 구현 |
| 점검일/현장 선택 | 점검 기준일과 현장 설정 | 구현 |
| 항목별 답변 | 항목별 답변 옵션 선택 | 구현 |
| 항목별 메모 | 특이사항 또는 조치 메모 입력 | 구현 |
| 점검 전/후 사진 | 사진 업로드 | 구현 |
| 저장 | 점검 이력 및 항목 결과 저장 | 구현 |
| 이력 조회 | 본인/현장 점검 이력 조회 | 신규 보정 완료 |
| 상세 확인 | 저장된 답변/사진 상세 확인 | 구현 |

### 6.3 주요 화면/API

| 구분 | 경로 |
|---|---|
| 작성 화면 | `/manage/daily-checks/form` |
| 이력 화면 | `/manage/daily-checks` |
| 이력 데이터 | `GET /manage/daily-checks/data` |
| 상세 | `GET /manage/daily-checks/{checkId}` |
| 체크리스트 항목 | `GET /manage/daily-checks/checklists/{checklistId}` |
| 저장 | `POST /manage/daily-checks/save` |
| 사진 포함 저장 | `POST /manage/daily-checks/save-with-photos` |
| 사진 조회 | `GET /manage/daily-checks/photos/{photoId}` |

### 6.4 정책

| 정책 | 내용 |
|---|---|
| 권한 | 현장 사용자는 본인 권한 현장의 점검만 작성 |
| 필수값 | 필수 항목 누락 시 저장 불가 |
| 사진 | 점검 전/후 사진을 점검 이력에 연결 |
| 이력 | 저장된 점검은 사용자와 관리자가 조회 가능 |

### 6.5 보완 필요사항

| 우선순위 | 항목 |
|---|---|
| 높음 | 모바일 화면 사용성 재검수 |
| 높음 | 조치 필요 항목의 후속 처리 상태 정의 |
| 중간 | 작성 후 수정 가능 여부와 이력 정책 |
| 중간 | 점검 결과보고서 양식과 점검일지 양식 분리 |

## 7. 관리자 > 일상점검 관리

### 7.1 목적

관리자가 현장 사용자가 등록한 일상점검 이력을 기간/현장/키워드로 조회하고, 상세 내용과 사진을 확인하며 보고서를 출력한다.

### 7.2 주요 기능

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 점검 목록 조회 | 기간, 현장, 상태, 키워드 검색 | 구현 |
