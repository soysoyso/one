# 정책 설계 초안

## 1. 보고서 정책

| 정책 ID | 정책 |
|---|---|
| REPORT-001 | 기존 PDF 다운로드 기능은 호환성을 위해 유지한다. |
| REPORT-002 | 신규 보고서 출력은 `template`과 `format`을 명시해야 한다. |
| REPORT-003 | 지원하지 않는 출력 형식 요청 시 사용자에게 명확한 오류를 반환한다. |
| REPORT-004 | DOCX는 Apache POI 기반으로 우선 구현한다. |
| REPORT-005 | HWPX는 확장 타입으로 예약하되, 초기 구현 범위에서는 스텁 또는 미지원 메시지로 처리한다. |
| REPORT-006 | 복합 보고서는 선택된 접수 건의 권한 검증을 모두 통과해야 생성한다. |

## 2. 일상점검 정책

| 정책 ID | 정책 |
|---|---|
| DAILY-001 | 사용 중인 체크리스트만 사용자 작성 화면에 노출한다. |
| DAILY-002 | 필수 항목이 누락되면 완료 처리할 수 없다. |
| DAILY-003 | 작성 완료 후에도 관리자는 수정 가능하되 수정 이력을 남기는 방향을 우선 검토한다. |
| DAILY-004 | 작업 전 이미지는 점검일지, 작업 후 이미지는 결과보고서에 우선 매핑한다. |

## 3. 상황일지 정책

| 정책 ID | 정책 |
|---|---|
| SITUATION-001 | 상황일지는 일자와 주/야간 구분을 필수로 한다. |
| SITUATION-002 | 상황정보는 시간순으로 정렬한다. |
| SITUATION-003 | 삭제는 물리 삭제보다 `use_yn = 'N'` 방식이 우선이다. |

## 4. 알림톡 정책

| 정책 ID | 정책 |
|---|---|
| NOTIFY-001 | 로컬 환경에서는 실제 발송 없이 수신자 설정 저장/조회만 검증한다. |
| NOTIFY-002 | 전화번호는 화면 표시와 저장 방식에서 개인정보 보호를 검토한다. |
| NOTIFY-003 | 알림 유형별 수신자를 분리한다. |

## 5. 권한 정책

| 정책 ID | 정책 |
|---|---|
| AUTH-001 | 관리자 설정 기능은 관리자 권한만 접근한다. |
| AUTH-002 | 현장 사용자는 본인 권한 현장의 일상점검만 작성한다. |
| AUTH-003 | 보고서 출력은 조회 권한이 있는 데이터만 허용한다. |
